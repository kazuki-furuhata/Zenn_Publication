## 2. 基調講演（Developer Keynote）：「Get real: Agents in the autonomous era」
4月23日 10:30 AM からスタートした Developer Keynote。

1日目の Opening Keynote が「What（何が出たか）」の場だったとすれば、こちらは完全に 「How（どう作るか）」 の場です。
お題は「ラスベガスで10,000人規模のマラソンをシミュレートするマルチエージェントシステムを、1時間でゼロから作る」。

機能紹介のスライドは少なく、複数の Google エンジニアがステージ上でライブコーディングしながら、エージェントを段階的に複雑化させていく——という演出の、かなり異色のキーノートでした。
途中でバグが発生し、そのまま本番デバッグするシーンもあり、デモながらも「これは現実の話だ」という信頼感が会場に漂っていました。

### 2.1 エージェントを「チームとして動かす」 ： マルチエージェント・オーケストレーション
最初のテーマは、単一エージェントから複数エージェントの協調へ。

Planner Agent（ルート設計）・Evaluator Agent（評価）・Simulator Agent（シミュレーション実行）という役割分担でシステムを構築。
このとき鍵となるのが以下の2つのコンポーネントです。

- **MCP（Model Context Protocol）**：
　Google が Linux 財団に寄贈した、エージェント間通信の標準プロトコルです。
　各エージェントは自分の能力を「Agent Card」として公開し、他のエージェントがそれを検索・発見して連携できます。
　「すべてのクラウドサービスが MCP を話す」という方向が示されており、APIの脆弱な結合を置き換える存在として位置づけられていました。

- **Agent Registry**：
　エージェントの「DNS」。誰が何をできるかを管理し、エージェントが自律的に必要なスキルを探してきます。
　Gemini Enterprise 上でユーザーが作ったエージェントも自動でここに登録され、組織全体で共有・再利用できる設計です。

また A2UI（Agent-to-UI） というオープン標準も登場しました。
エージェントが壁のようなテキストを返すのではなく、文脈に応じたインターフェースを動的に生成する仕組みです。
デモでは評価結果をエージェントが自ら視覚化していました。

### 2.2. 「記憶するエージェント」へ ： コンテキストエンジニアリングと Memory Bank
次のテーマは、ステートレスからステートフルへの転換です。

「シミュレーションの学習を次の計画に活かしたい」という要求に対し、過去のセッション履歴をそのままリクエストに詰め込む素朴な実装では、コンテキストウィンドウ（1Mトークン）の上限を超えてエージェントがクラッシュするという問題が発生しました。
解決策として示されたのが2つのアーキテクチャです。

- **Agent Platform Sessions**：
エージェントに文脈を持続させる仕組み。
20行以下のコードで実装できるデモは素直に驚きました。

- **Memory Bank**：
過去の実行から「学習」を長期記憶として保存・引き出す仕組み。
次回の計画立案に過去の知見を自動で組み込めるようになります。

さらに、ラスベガス市・ネバダ州の大会ルールを記載した非構造化ドキュメントを Data Engineering Agent がパイプライン化し、Auto-embeddings（BigQuery が自動でベクトル化）で RAG として活用するデモも披露されました。
1日目の Knowledge Catalog の話が、ここで具体的な実装として繋がってきた、という感覚がありました。

### 2.3. 「シフトダウン」で守る ： Zero-Trust Security in the Agentic Era
最後のテーマが、エージェント時代のセキュリティです。

デモで印象的だったのが、登壇者がプランナーエージェントに「予算を増やして」と指示したところ、ゼロトラストポリシーによって 書き込みがブロックされるというシーン。
登壇者が「グローストックとサングラスは承認できません。ゼロトラストにはゼロ予算ポリシーがあります」と笑いを取りながら、ガバナンスの重要性を示していました。

ここでのキーメッセージが 「Shift left ではなく Shift down」 です。
セキュリティの責任を開発者に押し付ける「Shift left」は持続可能ではない。プラットフォームレイヤーに落とし込むべきだ、という考え方です。

具体的には以下の仕組みで実現します。

- **Agent Identity**：
エージェントごとに固有の暗号IDを発行。
「サービスアカウントは全員共用の合鍵、Agent Identity はドアごとの生体認証」という説明が的を射ていました。

- **Agent Gateway**：
エージェント間通信のプロキシとして、IAM ポリシーを一元強制。

- **Wiz 連携（Red Agent / Green Agent）**：
Red Agent（AI の攻撃者役）が脆弱性を発見 → Green Agent（修正役）がコードを直接修正して CI/CD にコミット。
「開発速度とセキュリティをトレードオフにしない」という姿勢が明確でした。

「エージェントは賢いが、壊れ方も新しい」——そのリアルを正面から扱う姿勢に、去年よりも業界やAI界隈の成熟を感じました。

### 3.1 Define once, trust everywhere: The semantic layer for governed AI
1日目の基調講演や Knowledge Catalog のセッションを通じ、「セマンティックレイヤーをどこで作り込むか」という問いが浮かび上がっていましたが、このセッションがその最も直接的な回答でした。

セッションの入り口はシンプルな問いです。
**「売上」という言葉、マーケティング・営業・財務・カスタマーサクセスはそれぞれ同じ定義で使っていますか**？

よくある実態は「総売上 - 返品 = 純売上」という計算式が、部門ごとに微妙にズレている状態です。
そのズレが積み重なって下流の指標にも伝播し、役員会議で「この数字どっちが正しいの？」が発生する。
エージェントが増えれば、この問題は指数関数的に拡大します。

セマンティックレイヤーの役割は、この翻訳を一元化することです。
ビジネス側の言葉をデータの言語に変換する層を組織として1つ持つ、という考え方です。

Gemini が「月次アクティブユーザーは？」という質問に答えるとき、SQL をゼロから生成する方法と、LookML に定義済みの指標を API 経由で呼び出す方法では、精度と一貫性に大きな差が出ます。

前者は LLM が毎回「月次アクティブユーザー」を再解釈するため、定義がブレる。
後者は Looker に登録された自社定義の指標をそのまま使うため、定義がブレない。

「Define once, trust everywhere（一度定義して、どこでも信頼する）」というタイトルはこの設計を指しています。
このセッションで最も重要な設計思想がここでした。

LookML はコードベースなので Git 管理・CI/CD に乗せられ、バージョン管理も効く。
エンジニアリングの作法でデータの定義を管理できる点も強みです。

デモでは、既存の Looker Explore（売上・顧客データ）と、担当者のスプレッドシート（NPS データ）を組み合わせたエージェントを5分以内に構築していました。
「上位100顧客の NPS はいくつか」という問いに対し、エージェントが2つのデータソースを跨いで回答。

NPS 58 対 ワースト100顧客の -20、という数字を出しながら、計算ロジックも明示する。
CMO が知りたいことをチャットで聞ける状態を、アナリストではなくデータの仕組みで実現するデモでした。

Overdose Digital（EC 特化のグローバルエージェンシー）の事例も紹介されました。
Shopify・GA4 など複数ソースのデータを統合し、「どのクリエイティブがどの地域で粗利 71% の商品を引き上げているか」を分析→次の広告を自動生成→A/B テスト→デプロイまでエージェントが自律的に回すシステムです。
「データから行動へ、行動から学習へ」というループが完成している事例でした。

また Looker が MCP サーバーとして機能するようになり、任意の AI クライアントから LookML モデルを呼び出せるようになりました。
BigQuery にデータを置き、LookML で業務定義を登録し、Knowledge Catalog が集約してエージェントに渡す。
1日目から続いていた問いの構造が、ここで一本の線として繋がりました。

### 3.2 Accelerating Alpha: Citadel's hybrid HPC and AI strategy
「定量金融のハイステークスな世界では、インサイトに到達するまでのスピードこそが究極の競争優位だ」——この一文がこのセッションのすべてを表しています。

世界最大級のヘッジファンド Citadel Securities と Google Cloud が、HPC（高性能コンピューティング）× AI の最前線を語ったセッションです。
業界は金融ですが、内容は「スケールを伴う計算をどうクラウドで回すか」という普遍的な問いで、エンジニアリング的な示唆は業界を問いません。

Citadel がクラウドへ移行した出発点は、シンプルな物理的限界でした。
シカゴ本社ビルの電力供給が限界に達し、オンプレミスの拡張が不可能になったからです。

しかしより本質的な理由として語られていたのは、リソース使用のピーク量が予測できないという問題です。
量的研究のワークロードは不規則に急増する。18ヶ月先の計算需要は誰にも読めない。ピーク需要に備えてオンプレミスに投資し続けると、平時はアイドル状態のリソースだらけになる。
その「待ち時間」を消したい、というのがクラウド移行の核心的な動機でした。

「リフト＆シフトをやるだけでは意味がない、モダナイゼーションをセットでやる必要がある」という指摘も印象的でした。
移行の理由を明確にしないまま動かすだけでは、クラウドの価値は出ない、という話です。

アルファ生成ワークロードの大半を TPU に移行した結果、他のアクセラレーターと比較して 2〜4 倍の高速化 と 総所有コスト 30% 削減 を同時に実現しました。
「より速く、より安く」が両立した、というのは素直に驚きます。

TPU がこれほど効くのは、深層学習の行列演算に特化した設計（シストリックアレイアーキテクチャ）と、専用の光回路スイッチにより複数の TPU ポッドが「単一のスーパーコンピューター」として振る舞えるためです。
JAX との親和性も高く、1台から数千台へシームレスにスケールする設計になっています。

以下がCitadel が10年の経験から語った実践的な4つの教訓です。

- **「なぜ移行するか」を先に決める**：スケーラビリティなのか、コストなのか、目的によって戦略が変わる
- **段階的に進める**：一度にすべてを移行せず、成功基準を定義してから動く
- **ユーザーがクラウドを意識しない抽象化を作る**：研究者は「ジョブを投入する」だけで、インフラは自動でスケールする設計に
- **コストをビジネス成果と紐づける**：追跡ではなく予算執行管理。毎年「15% 高速化・15% コスト削減」という目標を継続的に追う

金融機関ならではの話として、セキュリティ設計の考え方も語られました。
「クラウドでオンプレミスで解決済みの問題を再度解決しない」という原則です。

コンプライアンス境界や情報バリアはオンプレミスを真実の情報源として維持し、新しいアカウントはオンプレミスで承認後にクラウドへ自動同期する。
この設計により、クラウドでのコンプライアンス対応がほぼ「非課題化」できたそうです。

「ピーク需要は予測できない」という現実から出発したクラウド移行が、TPU の活用と適切な抽象化設計によって研究速度・コスト・セキュリティのすべてで成果を出した事例でした。
業界が金融であっても、「インフラが研究速度に追従する体制を作る」という思想は、データ活用に取り組むあらゆる組織に刺さります。

### 3.3 Beyond the hype: How leading businesses win with Gemini Enterprise app
初日の基調講演を受け、「ユースケース集めと、お客様に『業界慣習ごと AI でやってみましょう』と言えるようになることが重要」という感想を持ちましたが、このセッションはまさにそのための素材でした。
Macquarie Bank（マッコーリー銀行）とドイツ証券取引所という2社の事例を軸に、Gemini Enterprise の「ハイプを超えた現実」を語るセッションです。

冒頭で示された数字が刺さりました。エンタープライズ AI の導入に成功しスケールできている企業は、業界全体でまだ 11% に留まっているという現実です。
残り 89% が失敗しているわけではなく、課題は明確で共通しています。

- 個人ではAIを使っているが、**職場ではチャットボット止まり**
- IT リソースが薄く、**全社展開を支える体制**がない
- セキュリティと利便性のバランスで**意思決定が止まる**
- 成果を**定量的に測る仕組みがない**

ツールがあるかどうかではなく、ガバナンス戦略と組織的なアプローチがあるかどうか。
これが導入成否を分ける、というメッセージは耳が痛いながらも正直な整理でした。

事例①：Macquarie Bank — **7ヶ月で 130,000 時間の生産性向上**
全行員 5,000 名に Gemini Enterprise を展開したマッコーリー銀行の事例です。
導入からわずか 7ヶ月で 130,000 時間 の生産性向上を記録。

印象的だったのは、アクセス権を一時的に失った従業員が「インターネットのない生活のようだ」とコメントしたというエピソードです。
ツールの導入ではなく、業務の本質的な変革が起きていることを示す言葉だと感じました。

特筆すべきは、IT 部門ではなく 法務・マーケティング・コンプライアンス・営業などの現場部門が自ら課題を解決し始めたという点です。
ハッカソンを開催したところ、数百のソリューションが生まれた。シニアリーダーの 97% が Google の GenAI リーダー認定を取得（30年以上試験を受けていない層を含む）という話も、組織全体の変容として面白いデータでした。

事例②：ドイツ証券取引所 — **3ヶ月の業務を 16 時間に**
日々数兆ユーロの取引を処理するドイツ証券取引所は、エラーが許されない高度規制環境です。
転機は「IT 開発は 6〜18 ヶ月かかるが、エージェントなら日単位で実装できる」と気づいたこと。

IT 部門の速度に頼るのではなく、課題を抱える現場部門に直接ツールを渡す戦略にシフトしました。
結果、約 100 種類の手数料に関する要件書の作成——従来は 10 名で 3 ヶ月 かかっていた業務が、Gemini 活用により品質は維持したまま 16 時間 で完了しました。

また、マルチエージェントの活用として「支払いを実行するエージェント」と「支払いが元の注文と一致しているか確認するカウンターエージェント」を並走させる品質管理フローも紹介されました。
金融機関ならではの「ダブルチェックをエージェントで実装する」という発想です。

プロダクトとしての説明も整理されていました。
Gemini Enterprise の位置づけは、組織の AI 活用への 「入口」 です。

新機能として紹介されたのは、以下の3つです。

- Projects : チームで成果物・ドキュメント・エージェントを共有する統一スペース
- Canvas : Gemini 内でスライド・ドキュメント・スプレッドシートを生成し Google Workspace / Microsoft に直接エクスポート
- Agent Designer : ビジュアルでエージェントを構築、MCP・サブエージェント・人間承認フローを追加可能

エージェント型の知能とif-then-else の決定論的ワークフローを組み合わせられる点が、従来の AI ツールとの明確な差別化です。
「ハイプを超えた現実」は、数字として存在していました。

130,000 時間、16 時間、97%——これらを生み出したのはツールではなく、現場部門へのエンパワーメントと、ガバナンスをセットで整えた組織的なアプローチです。
お客様への提案文脈で「何から始めるか」を語るときに、この2社の事例は使い勝手が良いと感じました。

## 4. まとめ：「作れる」から「運用できる」へ
Google Cloud Next '26 の2日目は、エンジニアにとって現実を突きつける1日でした。

1日目が「AI 前提の企業を作る」というビジョンを語ったとすれば、2日目は「では実際にどうコードを書き、どう動かし続けるか」という問いに正面から向き合っていました。
そしてこのキーノートを見て、もう一つ感じたことがあります。

非エンジニアが、AI とエージェントでエンジニアリングできる時代が来た。
では、エンジニアはどこに向かうのか。

Agent Designer でエージェントを設計し、Gemini Enterprise でノーコードのサブエージェントチームを作り、Cloud Assist に自然言語でインフラ変更を指示する。
今日のデモは、手を動かす力（実装力）がなくても、それに相当することができてしまうことを示していました。

ただし、それがそのまま技術力が不要になるを意味しないとも感じています。
むしろ逆です。「技術力」——つまりトレードオフを読み、事業や経営の判断材料に翻訳する力——は、今まで以上に問われる時代になる。 

エージェントが提案した設計を「なぜ良いか・どこが危ういか」と評価できるのは、依然として人間の技術的な思考力です。
実装の民主化が進むほど、判断の質が差になる。そんな時代の入口に立っている気がした2日目でした。
